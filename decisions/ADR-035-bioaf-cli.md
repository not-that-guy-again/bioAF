# ADR-035: bioaf CLI for In-Session Provenance and Heartbeat

**Status:** Accepted
**Date:** 2026-03-22
**Deciders:** Brent (repository owner)

---

## Context

bioAF tracks provenance for pipeline runs automatically: Nextflow logs capture every tool version, parameter, and input/output file. But ad-hoc analysis work on notebook sessions and work nodes (ADR-034) has no automatic provenance. When a comp bio SSH's into a work node, runs a script, and produces a figure, there is no record of what produced that figure -- which packages were installed, which data files were read, or which parameters were used.

Enforcing automatic provenance capture inside an SSH session would require invasive monitoring (shell hooks, filesystem watchers, process tracing) that conflicts with the platform's goal of providing flexibility without creating a walled garden. The alternative is voluntary provenance: a lightweight tool that makes it easy to snapshot the current state on demand, so users who want reproducibility can capture it with a single command.

This tool also serves a second purpose: heartbeat reporting for idle timeout detection on work nodes (ADR-034). Rather than building two separate agents, a single CLI tool handles both.

---

## Decision

Ship a `bioaf` CLI tool pre-installed in every compute environment image. The CLI is loosely coupled to the platform -- it knows only the bioAF API URL and its own hostname. All session context (project, experiment, environment, user) is fetched from the API at runtime based on the hostname.

### Architecture

```text
User runs: bioaf snapshot save "description"
                |
                v
  CLI calls GET /api/v1/compute-sessions/context?hostname=$(hostname)
                |
                v
  Backend looks up Pod by hostname, returns:
    { session_id, user_id, project, experiments,
      environment_name, environment_version,
      mounted_directories }
                |
                v
  CLI collects local state:
    - pip freeze (Python packages)
    - conda list (if conda is available)
    - R sessionInfo() (if R is available)
    - file listing of working directories
    - git status + git log (if in a git repo)
    - hostname, uptime, CPU/memory info
                |
                v
  CLI merges remote context + local state into a single JSON blob
                |
                v
  CLI POSTs to /api/v1/compute-sessions/{session_id}/snapshots
    { description, context_blob, local_blob }
                |
                v
  Backend writes to session_snapshots table,
  uploads full blob to GCS at snapshots/{session_id}/{snapshot_id}.json,
  emits SNAPSHOT_CREATED event to audit log
```

### Authentication

The CLI authenticates via a scoped API token mounted as a file at `/etc/bioaf/token` when the Pod is created. The token is generated by the backend at session launch time, scoped to the specific session (can only read context and write snapshots for that session), and expires when the session is terminated.

The CLI reads the token from the file. No interactive login, no user credentials stored on the node.

### CLI Commands

| Command | Purpose |
|---|---|
| `bioaf snapshot save "description"` | Capture environment state and upload to platform |
| `bioaf snapshot list` | List previous snapshots for this session |
| `bioaf env info` | Show current environment name, version, and image URI |
| `bioaf status` | Show session info: project, mounted data, uptime, quota usage |

All commands are read-only except `snapshot save`. The CLI never modifies local files, environment variables, or system state.

### Offline Fallback

If the bioAF API is unreachable (network issue, platform downtime), `bioaf snapshot save` writes the local state blob to `~/.bioaf/snapshots/{timestamp}.json` on disk and prints a message indicating the snapshot was saved locally. When connectivity is restored, the user can run `bioaf snapshot push` to upload pending local snapshots.

### Snapshot Data Model

```text
session_snapshots
  id, session_id (FK compute_sessions), description (text),
  context_json (JSONB -- project, experiment, env, mounts),
  local_json (JSONB -- packages, files, git status),
  gcs_uri (text -- full blob in GCS),
  created_by (FK users), created_at
```

The `context_json` and `local_json` columns store summary data for querying and display. The full blob (which can be large if file listings are extensive) is stored in GCS and referenced by `gcs_uri`.

### Heartbeat (Work Node Idle Detection)

The CLI also runs as a background service on work nodes, started by the Pod's startup script:

```bash
bioaf heartbeat --interval 300 &
```

Every 5 minutes, the heartbeat process:

1. Checks `/var/run/utmp` for active SSH sessions
2. Checks for user-owned processes with recent CPU activity (catches long-running computations in detached `tmux` sessions)
3. If activity is detected, POSTs to `POST /api/v1/compute-sessions/{session_id}/heartbeat`
4. If no activity is detected, skips the POST (the backend interprets missing heartbeats as idle)

The heartbeat uses the same scoped token as the snapshot commands. It runs silently and does not interfere with the user's work.

### Package and Distribution

The CLI is a small Python package (`bioaf-cli`) with minimal dependencies (only `urllib` from stdlib for HTTP, no third-party packages). It is installed in every environment image via a `RUN pip install bioaf-cli` line appended to the user's Dockerfile during the Cloud Build process (ADR-033). This injection happens automatically -- users do not need to include it in their environment definition.

The package is published to a private PyPI-compatible registry in Artifact Registry alongside the container images. Updates to the CLI are picked up when environment versions are rebuilt.

### Coupling Boundaries

The CLI is intentionally decoupled from the platform's internals:

- **No database models.** The CLI never imports bioAF backend code.
- **No knowledge of data model.** It receives and submits JSON blobs. The schema of those blobs is defined by the API, not hardcoded in the CLI.
- **No configuration beyond two values.** `BIOAF_API_URL` (where to send requests) and `/etc/bioaf/token` (how to authenticate). Both are injected by the platform at Pod creation time.
- **Works without the platform.** If the API is unreachable, the CLI degrades gracefully to local file output.

---

## Consequences

**Positive:**

- Voluntary provenance capture with a single command -- low friction, high value for reproducibility
- The CLI is thin and loosely coupled; it does not constrain the platform's API evolution
- Heartbeat reporting reuses the same tool, avoiding a separate monitoring agent
- Offline fallback means snapshots are never lost due to transient connectivity issues
- Automatic injection into all environment images means users never need to install it manually

**Negative:**

- Provenance is voluntary. Users who do not run `bioaf snapshot save` have no record of their ad-hoc analysis state. The platform cannot enforce this without becoming invasive.
- The `pip install bioaf-cli` injection adds a small amount of build time to every environment image and may conflict with environments that pin conflicting stdlib versions (unlikely given the zero-dependency design).
- The heartbeat's CPU activity check (for detached tmux processes) requires parsing `/proc` on Linux, which adds a small amount of system-level code to maintain.

**Neutral:**

- Snapshot data grows over time but is bounded by usage patterns (users save snapshots manually, not automatically). Storage costs are negligible.
- The scoped API token model means each session has its own credential. Token management is handled entirely by the platform; users never see or manage tokens.
